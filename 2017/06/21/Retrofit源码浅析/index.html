Retrofit伪动态代理

动态代理 是代理模式中的一种。网上很多文章都提到，Retrofit的接口封装HTTP API及隐藏OKHttp对网络请求的具体实现逻辑，就是利用的动态代理模式。然而，事实上，Retrofit利用的仅仅是动态代理的一个壳子，它在具体实现上与典型的动态代理有着根本的区别。

首先回到我们的 Demo，接口GitHubService事实上就是被代理类接口。它声明了一个listRepos()方法。然而 Demo 中我们并不用去实现这个接口，对动态代理熟悉的同学此时可能已经发现有些蹊跷，没有被代理类，如何利用 Proxy 去invoke被代理类的指定方法呢？别急，我们再往下看。

Retrofit Builder 建造了一个 Retrofit 对象，这里很简单，利用了Builder内部类形式的建造模式。建造模式 适用于初始化对象时需要大量可选方法的的情况，在android中有着广泛的使用。我在方法链中加入两个添加特定 Factory 的方法，之后会作讨论。

再回到Retrofit的代码中，与传统动态代理比较，不难发现，我们并没有得到被代理类对象(因为我们压根儿没有定义GitHubService的实现类),更别提去invoke()其方法了，那么这里的InvocationHandler.invoke()究竟做了什么呢？答案就在代码第十九行：

ServiceMethod类对象同样通过内置Builder建造，注意四五六这三行代码，ServiceMethod储存了method的方法注解表、参数表以及参数注解表。是不是有些头绪了呢？回到 Demo 中的 HTTP API 接口 listRepos()方法，当此方法通过GitHubService对象被调用时(InvocationHandler.invoke()方法随即被调用)，Retrofit关心的根本不是它的实现，而是它的方法注解表、参数表以及参数注解表。这也正是我所说的，Retrofit伪动态代理的原理，利用动态代理的壳子，得到了自己真正想要的东西。

链式调用巧妙连接RxJava

如果你看过Retrofit1的源码，就会发现Retrofit.create()方法和现在有所不同，最主要的差别就是InvocationHandler.invoke()原先直接返回一个 Call 对象，而现在则是返回了这样一个东西：

return serviceMethod.callAdapter.adapt(okHttpCall);

首先，我们先来假设一下，之前说到过，在接口GitHubService调用listRepos方法时，实际上会执行InvocationHandler.invoke()方法，而最开始我在 Demo 中也提醒大家，我将会把 listRepos 方法的返回对象由 Call 改为 Observable，那么不难推测，这个serviceMethod.callAdapter.adapt()方法很有可能就是返回一个 Observable 对象。按照这个思路，我们来看一看源码。
